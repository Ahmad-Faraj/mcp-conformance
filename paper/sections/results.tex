\label{sec:results}

We probed \Nprobed{} servers---the complete eligible frame, a census rather than a
sample. We organize results around the five research questions: runnability (RQ1),
protocol conformance (RQ2), robustness (RQ3), spec--practice divergence and its cause
(RQ4), and correlates (RQ5).

\subsection{RQ1: Runnability and the startup funnel}
Figure~\ref{fig:funnel} shows the funnel from eligible server to completed handshake.
Of \Nprobed{} servers, \HandshakeCount{} (\HandshakeRate{}) completed the MCP
handshake. The remaining two-in-five are classified by cause in
Table~\ref{tab:startup}: launch crashes (by exit signature), silent exits, hangs,
credential/configuration requirements, and package-install failures. Separating the
credential/configuration class is essential: without it, servers that merely need
keys would be conflated with servers that genuinely crash. Strikingly, a large share
of failures occur \emph{after} a successful install---servers that a client can
fetch but that then crash, hang, or exit without serving the protocol---so the
result is not merely an artifact of broken packaging. The startup-failure taxonomy
is a contribution in its own right: no prior work has measured how often
registry-published servers fail to start, and because we census the entire eligible
frame, these are population counts rather than estimates.

\subsection{RQ2--RQ3: Conformance and robustness}
Table~\ref{tab:verdicts} reports the verdict distribution for each check over the
\Nresponders{} responding servers, with 95\% Wilson confidence intervals. Lifecycle

\MalformedDiesRate{} crashed or hung on a malformed frame---the latter an
availability risk we treat under responsible disclosure.

\subsection{RQ4: The spec--practice divergence and its cause}
\ErrAsResultRate{} of responding servers reported a call to a non-existent tool as
an \texttt{isError} tool-execution result rather than the JSON-RPC protocol error the
specification illustrates. As established in Section~\ref{sec:threats}, we treat this
as a divergence rather than a violation, because the specification uses no RFC-2119
keyword here. To explain its near-universality, we attribute each server to its SDK
from package metadata (Table~\ref{tab:sdk}). The behavior tracks the SDK: the
official TypeScript and Python SDKs and the popular FastMCP framework all default to
it, and even most hand-rolled servers follow suit. A small number of
TypeScript-SDK servers emit the protocol error, showing the conforming behavior is
achievable within the SDK but is not its default. We confirm this directly against
the official reference servers, which exhibit the same divergence. The finding is
therefore not that thousands of authors independently erred, but that a handful of
SDK defaults set ecosystem behavior.

\subsection{RQ5: Correlates}
We test whether runnability and conformance correlate with the package registry
(Figure~\ref{fig:byregistry}). Runnability differs sharply: npm servers complete a
handshake far more often than PyPI servers ($65.9\%$ of $3{,}968$ vs.\ $50.0\%$ of
$2{,}138$; two-proportion $z=12.1$, $p<10^{-3}$). Conformance, by contrast, is
uniformly poor across both registries---the error-as-result divergence appears in
$87.7\%$ of responding npm servers and $90.2\%$ of responding PyPI servers---which
is consistent with RQ4: because the divergence is inherited from the SDKs, and both
ecosystems' SDKs share the same default, the registry a server ships on changes
whether it \emph{starts} but not whether it \emph{conforms}. The runnability gap is
a genuine ecosystem-quality signal and not an artifact of the credential/configuration
class, which we account for separately in the funnel.
